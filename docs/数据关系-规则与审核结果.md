# 数据关系：规则库与审核结果

本文说明当前前端 mock 实现里，**规则配置**与**单次合同审核结果**分别存放在哪里、类型如何对应、以及如何对照阅读代码。不涉及后端接口约定。

---

## 1. 两套主数据的位置

| 数据 | 含义 | 代码位置 | 在界面中如何查看 |
|------|------|----------|------------------|
| **规则库** | 规则名称、所属模块、状态、双提示词、字段模板、版本历史等 | `src/pages/RulesPage.tsx` 中导出的 **`mockRules`**，单条类型为 **`MockRule`** | 打开 **规则列表**；进入 **规则详情** 路由 `/admin/rules/:ruleId`，详情页通过 `ruleId` 在 `mockRules` 中 `find` |
| **本次评审结果** | 某份合同一次审核产生的多条「风险 / 识别」卡片 | `src/pages/ReviewerPage.tsx` 中 **`useState<RiskItem[]>([...])` 的初始数组** | 打开 **审核页**，右侧「规则评审结果」列表即 `riskItems` 的展示；统计、四颗胶囊、待处理跳转等均基于 `riskItems` 及其派生数据 |

---

## 2. 类型与业务含义的对应关系

### 2.1 输出模式（规则侧 ↔ 结果侧）

- **`MockRule.ruleMode`**（类型 `RuleOutputMode`，定义于 `RulesPage.tsx`）
- **`RiskItem.ruleMode`**（类型 `RuleMode`，定义于 `ReviewerPage.tsx`）

二者枚举一致：

| 值 | 含义 |
|----|------|
| `bpm_structured` | 评审结果：可展示结构化字段、采纳/忽略等，参与审核点结构化回传（产品语义） |
| `recognition_only` | 仅识别：以识别摘录等为主，不向 BPM 回传结构化审核点字段 |

### 2.2 风险维度 category（仅结果侧每条命中）

- **`RiskItem.category`**：`trigger`（触发风险）、`missing`（信息缺失）、`low`（规则内 / 低风险）。
- **`MockRule` 本身不存储 category**：category 是模型对**某次合同**的判定输出；规则列表里若展示三分类，使用的是同一套 **`RuleCategory`** 语义（见 `RulesPage.tsx` 中 `categoryMeta`），与审核页胶囊一致。

### 2.3 规则 ID 与结果条目的关联（当前 mock 现状）

- **`RiskItem` 当前没有 `ruleId`（或等价字段）**。
- 因此 **`mockRules` 中的 `R001`、`R002` 等与 `ReviewerPage` 里 `risk-1`、`risk-2` 等**在代码层面没有外键关联**；演示场景仅通过名称、业务描述等**人工对齐**。
- 若产品需要「本条结果来自哪条规则」，建议在 `RiskItem` 上增加 `ruleId`，并在 mock 数据中显式填写对应关系。

## 3. RiskItem 结构要点（审核页一条卡片）

接口定义见 **`src/pages/ReviewerPage.tsx`** 中的 **`RiskItem`**。

### 3.1 与左侧合同定位的关系

以下字段中的 id 需与左侧合同段落 DOM 的 id 对齐（`collectReferencedLocationIds` 会汇总引用）：

- `locationId`、`reasonLocationId`
- `structuredFields[].locationId`
- `recognitionClauses[].locationId`（仅识别模式）

### 3.2 影响「待 / 已 / 忽」与交互的状态字段（结构化模式）

在 **`ruleMode === 'bpm_structured'`** 前提下：

- **`isAccepted`**：采纳（参与 BPM 语义）。
- **`acceptedAfterEdit`**：编辑后确认，与采纳效果一致，仅文案区分。
- **`ignored`**：忽略，与采纳互斥。

统计与「待处理上一条/下一条」等逻辑仅针对结构化且未忽略、未采纳的条目（具体条件以 `ReviewerPage.tsx` 内实现为准）。

### 3.3 仅识别模式

- **`recognitionClauses`**：优先于 `structuredFields` 的展示形态。
- **`recognitionFeedback`**：赞/踩等反馈，不参与 BPM 结构化回传。

## 4. 推荐阅读顺序（读代码）

1. **`RulesPage.tsx`**：`MockRule`、`mockRules`、`categoryMeta`、`ruleOutputModeMeta`。
2. **`RuleDetailPage.tsx`**：如何根据路由参数加载单条规则。
3. **`ReviewerPage.tsx`**：`RiskItem`、`RuleMode`、`StructuredField`、`RecognitionClauseItem`，以及 `riskItems` 初始数据与各类 `setRiskItems` 更新逻辑。
